= BOSS GX-700 graphical editor — roadmap & ideas
:toc: left
:icons: font
:source-highlighter: rouge

Planning notes for a *graphical patch editor* for the BOSS GX-700 (Rackctl Stage 4).
*Not built yet* -- today the GX-700 has a library (`crates/gx700`) and a CLI
(`rackctl-gx700`). This document records the feature ideas and how they map onto the
building blocks that already exist, so the GUI can be built deliberately when the
time comes.

== Headline feature: the patch librarian & level-balancer

A one-screen view of *all patches* that makes it easy to *balance their volumes* --
the most-requested idea, and a real pain point (a typical bank has output levels all
over the place: we have seen 22 %, 50 %, 80 %, 100 % in one user bank, which is
awkward live).

The view:

* A *list of every patch* (user `U001`..`U100`, and presets `P001`..`P100`), each
  row showing the patch name and its *output level* ("volume").
* *One click selects a patch and plays it* so you can hear it.
* The output level is *editable inline* (a slider / number per row) so you can nudge
  patches until their perceived loudness matches across the bank.

=== Building blocks that already exist

[cols="1,3",options="header"]
|===
| Need | Already have
| The list (name + level + chain per slot) | `Gx700::read_patch_header` -- the CLI `patches` command renders exactly this.
| Click-to-hear a patch | `preview <slot>` writes the patch into the *active sound*, non-destructively, and works in *any* mode (better than Program Change, which the GX-700 ignores in BULK LOAD mode).
| The "volume" control | The `output-level` parameter (Level/Chain offset `00`, raw `0..100`, shown as a percentage).
| Editing a level live | `set output-level …` (an individual temporary-buffer write; instant, no special mode).
| Storing an adjusted level | A whole- or part-patch memory write -- see the mode wrinkle below.
|===

=== The one wrinkle: live vs. stored (BULK LOAD)

There are two flavours of "set the volume", and they differ by device mode (see
`gx700-sysex-protocol.adoc`):

* *Live* -- changing the *active sound's* level is instant and needs no special mode.
* *Persisting* a level into a *stored patch* is a memory write, which the GX-700
  accepts only while it is in front-panel *BULK LOAD* mode.

So the natural flow is *balance live, write once*: click through the patches
adjusting each one's level by ear, then a single *"Write levels to the unit"* action
that guides the user into BULK LOAD mode and stores the adjusted patches as a batch.
That keeps the per-patch BULK LOAD dance out of the everyday loop -- you do it once,
at the end.

== Other candidate features (from the Stage 4 plan)

* *Effect-chain editor* -- per-block enable + parameter widgets for the 14 effect
  blocks, and drag-to-reorder the signal chain (the library already has
  `RawPatch::chain` / `set_chain` and the CLI `chain` command).
* *Patch librarian* -- save / load / `backup` / `scene` from the GUI, reusing the
  US-16x08 GUI's preset-tab patterns.
* *Per-block response curves* -- reuse the US-16x08 GUI's `curves.rs` (EQ, etc.).

== Reuse

Per the Stage 4 plan, lift the *device-agnostic* parts of the US-16x08 GUI:
`poller.rs` (background reader + reconnect), `config.rs` (ProjectDirs settings +
migration), `preset_tab.rs` (generic preset rows), `curves.rs` (plotting). The new,
GX-700-specific UI is the patch list / level-balancer and the effect-chain editor.
App-id `rackctl-gx700-gui`; settings under `~/.config/rackctl/gx700/`.
