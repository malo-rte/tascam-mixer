This manual describes how to control the BOSS GX-700 guitar effects processor
from Linux using the command-line tool `rackctl-gx700`.

Read <<sec-install>> to build and install the tool and <<sec-connecting>> to wire
up and find the unit. After that, <<sec-patches>> covers browsing and recalling
patches, <<sec-editing>> covers changing a sound's parameters, and
<<sec-scenes-backups>> covers saving and restoring patches to your computer.
<<sec-monitor>> is a diagnostic aid, and the reference chapters
(<<sec-cli-ref>>, <<sec-troubleshooting>>, <<sec-glossary>>) are for looking up
details. Every command is also self-documenting with `--help`.

[[sec-intro]]
== Introduction

The GX-700 is a rackmount guitar effects processor. Its sounds are stored as
*patches*: 200 of them, in two banks of 100 -- the *user* patches (which you can
overwrite) and the *preset* patches (factory data). It has no mixer and no level
meters; it is edited entirely over MIDI.

`rackctl-gx700` reaches the unit over a 5-pin DIN MIDI connection through a
USB-MIDI interface (for example, the MIDI ports of a Tascam US-16x08). It can read
and write individual effect parameters, select patches, and -- the subject of this
manual -- save and restore whole patches and whole banks to your computer.

[[sec-install]]
== Installing the tools

[[sec-install-prereq]]
=== Prerequisites

* A Linux system with ALSA (any modern kernel).
* A *USB-MIDI interface* with 5-pin DIN MIDI in and out, to bridge the computer to
  the GX-700 (for example, the MIDI ports of a Tascam US-16x08).
* To build from source: a Rust toolchain (edition 2024, 1.95 or later) and the
  ALSA development headers, which the default build links against. On Debian or
  Ubuntu: `sudo apt-get install libasound2-dev`.

[[sec-install-build]]
=== Building from source

Build the command-line tool from the workspace root:

[source,console]
----
$ cargo build --release -p rackctl-gx700-cli
----

The binary is `target/release/rackctl-gx700`. Copy it onto your `PATH` (for
example `~/.local/bin/`), or run it in place. This manual writes it simply as
`rackctl-gx700`.

NOTE: The Nix flake in this repository currently packages the *US-16x08* tools; the
GX-700 CLI is built with `cargo` as above.

[[sec-install-mock]]
=== Running without hardware

Every command accepts a global `--mock` flag (used instead of `--port`) that runs
against an in-memory device with no interface attached. It is handy for trying the
commands, reading the catalog, and seeing the on-disk file layout. The mock starts
fresh each run, so it does not persist edits between commands.

[source,console]
----
$ rackctl-gx700 --mock list            # the parameter catalog
$ rackctl-gx700 --mock dump             # a (default) patch, in readable form
----

[[sec-install-completions]]
=== Shell completions

`rackctl-gx700 completions <shell>` prints a completion script to standard output
for `bash`, `elvish`, `fish`, `powershell`, or `zsh`. Redirect it where your shell
looks for completions, for example:

[source,console]
----
$ rackctl-gx700 completions bash | sudo tee /usr/share/bash-completion/completions/rackctl-gx700
$ rackctl-gx700 completions fish > ~/.config/fish/completions/rackctl-gx700.fish
----

[[sec-connecting]]
== Connecting to the GX-700

Connect the GX-700's MIDI IN and MIDI OUT to a USB-MIDI interface on your computer
(OUT-to-IN in both directions), then find the interface's port:

[source,console]
----
$ rackctl-gx700 ports
hw:1,0
----

That `hw:CARD,DEV` address is passed to every command that talks to the hardware,
with `--port`:

[source,console]
----
$ rackctl-gx700 --port hw:1,0 patches
U001  N RODGERS?    out 50%
U002  N RODGERS?    out 38%
...
----

[IMPORTANT]
====
*Only one program may use the MIDI port at a time.* The GX-700 replies to a
request on the same port the request went out on; if two programs read it at once,
each receives only part of every reply and both fail with a timeout that looks
exactly like a dead interface. `rackctl-gx700` takes a lock on the port, so a
second command fails fast with `MIDI port hw:1,0 is in use by another rackctl
process` rather than corrupting the first. While a backup or restore is running,
do not start another GX-700 command -- just let it finish.
====

TIP: Every command also runs against an in-memory mock device with `--mock` (and
no `--port`), so you can try the commands and see the file layout without the
hardware connected.

[[sec-patches]]
== Patches

A *patch* is one complete GX-700 sound -- every effect block and its settings,
plus a name and the order of the effect chain. The unit holds *200* of them in two
banks of 100: the *user* patches `U001`..`U100`, which you can overwrite, and the
*preset* patches `P001`..`P100`, which are factory data.

This chapter is about browsing the patches on the unit, recalling one, and viewing
one in detail. Editing the current sound is in <<sec-editing>>; saving patches to
your computer is in <<sec-scenes-backups>>.

[[sec-patches-list]]
=== Listing the patches on the unit

`patches` lists the user bank; `--preset` lists the factory bank. Each line is the
slot, the patch name, and its output level:

[source,console]
----
$ rackctl-gx700 --port hw:1,0 patches
U001  N RODGERS?    out 50%
U002  N RODGERS?    out 38%
U003  CLEAN CUTTIN  out 22%
...

$ rackctl-gx700 --port hw:1,0 patches --preset
P001  STUDIO SPACE  out 80%
P002  TUBE STACK    out 80%
P003  SPACE OCTAVE  out 80%
...
----

Listing a whole bank reads each patch in turn, so it takes a little while (it is
paced to keep the interface stable -- see the one-program-at-a-time rule in
<<sec-connecting>>).

To list the patches you have saved on your *computer* instead of on the unit, add
`--disk` (this needs no device); see <<sec-scenes-backups>>.

[[sec-patches-select]]
=== Recalling a patch

`select <n>` recalls a patch into the current sound by sending a MIDI Program
Change:

[source,console]
----
$ rackctl-gx700 --port hw:1,0 select 7
----

[NOTE]
====
`n` is a *Program Change number* (0--127), which is not necessarily the patch
number: which patch a Program Change recalls is governed by the unit's Program
Change map, a GX-700 system setting. Recalling a patch replaces the current sound,
so any unsaved edits to it (see <<sec-editing>>) are lost.
====

[[sec-patches-preview]]
=== Previewing a patch

`preview <slot>` writes a stored patch (by its *slot number*, 1--200, user or
preset) into the active sound so you can *hear* it, without storing it anywhere:

[source,console]
----
$ rackctl-gx700 --port hw:1,0 preview 112   # audition preset 12 (slot 112)
----

Unlike `select`, `preview` takes the actual patch slot (not a Program Change
number), and -- because it edits the active sound rather than recalling a memory --
it works *even while the unit is in BULK LOAD mode* (where Program Change does
nothing). It is non-destructive: nothing in memory changes; recall any patch (or
power-cycle) to restore the original sound. This is the way to audition presets one
by one -- e.g. before `copy`-ing a favourite into the user bank.

[[sec-patches-dump]]
=== Viewing a patch in detail

`dump` prints a patch in readable form -- the header, then every block with its
parameter values. It can read three sources:

`dump`:: the *current sound* (the temporary buffer).
`dump --patch N`:: a stored slot `N` (1--200) on the unit.
`dump --file NAME`:: a saved patch *file* on your computer (no device needed).

[source,console]
----
$ rackctl-gx700 --port hw:1,0 dump --patch 42
Patch: "CRUNCH RHY"
  output level: 80%
  chain: Compressor > Wah > Distortion > Preamp > Equalizer > Delay > Reverb
  [Compressor]
    comp-enable            off
    ...
  [Preamp]
    preamp-enable          on
    preamp-type            BG Lead
    preamp-volume          31
    preamp-bass            87
    preamp-middle          58
    preamp-treble          50
    preamp-presence        7
    preamp-master          95
    preamp-bright          off
    preamp-gain            mid
  ...
----

The header gives the name, the output level, and the *chain* -- the order the
effect blocks sit in the signal path, which a patch may reorder. Then each block
lists its parameters in display form: booleans as `on`/`off`, enums as their label,
integers as their number. A `-` means that block carries no data in this patch.

`dump --file` reads from your saved library -- the files written by `save`,
`backup`, and `scene save` (see <<sec-scenes-backups>>). For example, after a
`backup` you can read any patch back without the unit connected:

[source,console]
----
$ rackctl-gx700 dump --file U042
----

[[sec-bulk-load]]
=== Storing patches to the unit (BULK LOAD mode)

[IMPORTANT]
====
The GX-700 accepts writes to its patch memories *only while it is in BULK LOAD
mode*. Every command that stores a patch on the unit -- `copy`, `load --to-patch`,
and `scene restore` -- needs this one extra step on the hardware first:

. On the GX-700, press *TUNER/UTILITY*.
. Use the PARAMETER buttons to select *"MIDI BULK LOAD"*; the display shows
  *"Waiting..."*.
. Run the `rackctl-gx700` command. The unit's display shows "receiving...", then
  "idling...".
. Press *PLAY* on the unit to finish.

Outside BULK LOAD mode the write is silently ignored by the unit -- but the tool
reads every write back to check it, and tells you plainly if it did not store
("the GX-700 did not store the patch ... only in BULK LOAD mode ... re-run this
command"), so a forgotten step never looks like success. Reading, listing, backing
up, and editing the current sound do *not* need this mode.
====

[[sec-patches-copy]]
=== Copying a patch

`copy <from> <to>` duplicates a stored patch into another slot on the unit, without
going through a file. The source may be any patch -- a user patch (1--100) or a
preset (101--200) -- and the destination must be a user slot (1--100), which it
overwrites. The unit must be in BULK LOAD mode (see <<sec-bulk-load>>):

[source,console]
----
$ rackctl-gx700 --port hw:1,0 copy 12 42     # copy user patch 12 into user slot 42
$ rackctl-gx700 --port hw:1,0 copy 112 42    # copy preset 12 (slot 112) into user slot 42
----

The patch's name travels with it, so the destination ends up identically named. This
is the same as a `save` followed by a `load --to-patch`, in one step -- a handy way
to pull a favourite preset into the user bank where you can edit it.

[WARNING]
====
`copy` overwrites the destination patch; whatever was in that slot is lost unless
you have saved it (with `save`, `backup`, or `scene save`). To copy *into* a preset
slot is not possible -- the factory bank is read-only.
====

[[sec-patches-chain]]
=== Reordering the signal chain

A patch's *signal chain* is the order its effect blocks sit in the signal path.
`chain` shows or reorders that order. It works on a *saved patch file* (on disk, no
device); after reordering, `load` the file to apply it on the unit.

Show a saved patch's chain:

[source,console]
----
$ rackctl-gx700 chain "my-lead"
 1. Compressor [comp]
 2. Wah [wah]
 3. Distortion [dist]
 4. Preamp [preamp]
 5. Loop [loop]
 6. Equalizer [eq]
 7. Speaker Sim [speaker]
 8. Noise Suppressor [ns]
 9. Modulation [mod]
10. Delay [delay]
11. Chorus [chorus]
12. Tremolo/Pan [tremolo]
13. Reverb [reverb]
----

Reorder it with `--set`, giving a comma-separated list of all 13 block tokens (the
`[token]` shown above) in the order you want. For example, to put Delay *before*
Modulation:

[source,console]
----
$ rackctl-gx700 chain "my-lead" --set comp,wah,dist,preamp,loop,eq,speaker,ns,delay,chorus,mod,tremolo,reverb
----

The list must be a *permutation of all 13 blocks* -- each exactly once, none
repeated or omitted (a wrong list is rejected). All 13 blocks always stay in the
chain; to take one out of the sound, turn its `*-enable` off (see <<sec-editing>>)
rather than removing it here. `--set` saves the edited patch back to its file; then
write it to the unit:

[source,console]
----
# put the GX-700 in BULK LOAD mode (see <<sec-bulk-load>>), then:
$ rackctl-gx700 --port hw:1,0 load "my-lead" --to-patch 42
----

[[sec-patches-banks]]
=== User and preset banks

[cols="1,1,3",options="header"]
|===
| Bank | Slots | Notes
| User   | `U001`..`U100` (1..100)   | Editable. `copy`, `load --to-patch`, and `scene restore` store here -- in BULK LOAD mode (see <<sec-bulk-load>>). Overwriting is destructive.
| Preset | `P001`..`P100` (101..200) | Factory data. You can list, `dump`, and `backup --preset` them; the tools never write to them, so they are effectively read-only.
|===

[[sec-editing]]
== Editing parameters

A GX-700 sound is a chain of effect blocks -- a compressor, a wah, a
distortion, a preamp, an equalizer, modulation, delay, reverb, and so on.
`rackctl-gx700` reads and writes the individual parameters of those blocks, each
addressed by a short stable *key* such as `preamp-type`, `delay-time`, or
`output-level`.

[[sec-editing-model]]
=== What `get` and `set` act on

[IMPORTANT]
====
`get` and `set` operate on the *current sound* -- the temporary buffer, i.e. what
the unit is playing right now -- and a `set` takes effect *immediately* (you hear
it). They do *not* change a stored patch. Editing leaves the 200 patch memories
untouched until you explicitly save (see <<sec-editing-keep>>).
====

So the usual flow is: pick a starting patch, edit the current sound, listen, then
keep it if you like it.

[source,console]
----
$ rackctl-gx700 --port hw:1,0 select 42     # load user patch 42 into the current sound
$ rackctl-gx700 --port hw:1,0 set preamp-type "Metal 5150"
$ rackctl-gx700 --port hw:1,0 set preamp-volume 90
----

To read back a *stored* patch instead of the current sound, use `dump --patch N`
(a readable dump, no editing) or `select` it first so it becomes the current sound.

[[sec-editing-find]]
=== Finding a parameter

`list` prints the whole catalog -- 91 parameters across the 14 blocks -- as key,
block, kind, and patch offset:

[source,console]
----
$ rackctl-gx700 list
KEY                    BLOCK              KIND                     PATCH OFFSET
output-level           Level/Chain        int 0..=100              00 00 00 00
comp-enable            Compressor         bool                     00 00 01 00
comp-type              Compressor         enum[2]                  00 00 01 01
...
----

`info <key>` shows one parameter in detail -- its block, its kind, and its range
or enum values:

[source,console]
----
$ rackctl-gx700 info preamp-type
preamp-type  (Preamp)
  patch offset: 00 00 04 01
  live address: 08 00 04 01
  kind:  enum (default 0)
  values: 0=JC-120  1=Clean Twin  2=Match Drive  3=BG Lead  4=MS1959 (I)  ...
----

Neither command needs the hardware -- the catalog is built in -- so `list` and
`info` work without `--port`.

[[sec-editing-kinds]]
=== Kinds of value

Every parameter is one of three kinds (the `KIND` column, and the `kind:` line of
`info`):

Boolean:: An on/off switch -- a block's `-enable`, or a toggle like
`preamp-bright`. Accepts `on`/`off`, `true`/`false`, `1`/`0`, or `yes`/`no`.
`get` prints it as `true`/`false`.

Integer:: A number in a raw range, usually `0..=100` (`info` shows the range). Set
it with a plain number. A few are shown with units -- `output-level` reads back as
a percentage, for example.

Enum:: A choice from a fixed list, like `preamp-type`. Set it either by index
(`0`, `1`, `2`, ...) or by its case-insensitive label (`"BG Lead"`). `get` prints
it as `label (index)`.

[[sec-editing-read]]
=== Reading a value

[source,console]
----
$ rackctl-gx700 --port hw:1,0 get preamp-type
BG Lead (3)
$ rackctl-gx700 --port hw:1,0 get preamp-volume
90
$ rackctl-gx700 --port hw:1,0 get output-level
80%
----

[[sec-editing-write]]
=== Changing a value

`set <key> <value>` writes the value to the current sound. It is silent on
success; `get` to confirm.

[source,console]
----
$ rackctl-gx700 --port hw:1,0 set preamp-enable on           # boolean
$ rackctl-gx700 --port hw:1,0 set preamp-volume 90           # integer
$ rackctl-gx700 --port hw:1,0 set preamp-type "Metal 5150"   # enum by label
$ rackctl-gx700 --port hw:1,0 set preamp-type 8              # enum by index (the same thing)
----

Out-of-range or unknown values are rejected, and an enum error lists the valid
choices:

[source,console]
----
$ rackctl-gx700 --port hw:1,0 set preamp-volume 200
error: value 200 out of range for preamp-volume (expected 0..=100)
$ rackctl-gx700 --port hw:1,0 set preamp-type "Tweed"
error: unknown value "Tweed"; expected one of: JC-120, Clean Twin, Match Drive, BG Lead, ...
----

[[sec-editing-keep]]
=== Keeping an edit

Because edits change only the current sound, they are gone the next time you select
another patch -- unless you store them. Save the current sound to a file, then load
it into a user slot:

[source,console]
----
$ rackctl-gx700 --port hw:1,0 save "my-lead"                  # current sound -> file
# put the GX-700 in BULK LOAD mode, then:
$ rackctl-gx700 --port hw:1,0 load "my-lead" --to-patch 42    # file -> user slot 42 (destructive)
----

Writing to a stored slot needs the unit in BULK LOAD mode (see
<<sec-bulk-load>>); see <<sec-scenes-backups>> for saving and loading in full. (You
can also store the current sound from the GX-700's own front panel with its Write
button.)

[[sec-editing-blocks]]
=== The effect blocks

Each block has an `<prefix>-enable` on/off switch and a set of parameters; most also
have a `-type` choosing the model or mode. The keys group by block prefix:

[cols="1,1,1,3",options="header"]
|===
| Block | Key prefix | Params | What it is
| Level / Chain | `output-level` | 1 | Patch output level (the chain order and name travel with whole-patch save/load, not as individual parameters)
| Compressor    | `comp-`     | 8  | Dynamics compressor / limiter
| Wah           | `wah-`      | 12 | Wah, auto-wah, and pedal filters
| Distortion    | `dist-`     | 6  | Overdrive, distortion, fuzz
| Preamp        | `preamp-`   | 10 | COSM amp models and tone stack
| Loop          | `loop-`     | 4  | External effects-loop send / return
| Equalizer     | `eq-`       | 7  | Four-band EQ
| Speaker       | `speaker-`  | 5  | Speaker / cabinet simulator
| Noise         | `ns-`       | 5  | Noise suppressor / gate
| Modulation    | `mod-`      | 2  | Flanger / phaser / pitch / harmonist / ... (only on/off and type are catalogued so far; the per-type parameters are still being mapped)
| Delay         | `delay-`    | 11 | Delay
| Chorus        | `chorus-`   | 8  | Chorus
| Tremolo / Pan | `tremolo-`  | 5  | Tremolo and auto-pan
| Reverb        | `reverb-`   | 7  | Reverb
|===

TIP: `list` and `info` are the authoritative reference -- the table above is a map;
`info <key>` gives the exact range or the full list of enum labels for any
parameter.

[[sec-editing-mock]]
=== Trying it without hardware

`--mock` (with no `--port`) runs `set`/`get` against an in-memory device, handy for
checking command and value syntax. Note that the mock starts fresh on every run, so
a `set` in one command is *not* visible to a `get` in the next -- each invocation is
a new mock device seeded with defaults. On real hardware a `set` persists in the
current sound until you change it or select another patch.

[[sec-scenes-backups]]
== Scenes & backups

There are three ways to keep copies of your sounds on the computer, from a single
patch up to the whole device. They all write plain JSON files you can copy, keep
in version control, or move between machines.

[[sec-mental-model]]
=== Patches, scenes, and backups

Single patch:: One sound. Save it with `save`, bring it back with `load`. Good for
keeping a favourite tone or moving one patch between slots.

Scene:: A snapshot of the *whole device* -- all 100 user patches -- captured under
one name in a single file. Think of it as "everything in my GX-700 right now",
saved as one unit for a gig or a project. Save with `scene save`, restore with
`scene restore`.

Backup:: A *flat copy of every patch as its own file* (`U001.json` ... `U100.json`),
written into your patch library so you can inspect or restore any one of them by
name. Save with `backup`.

[NOTE]
====
*Scenes and restores cover the 100 user patches only.* The 100 preset patches are
factory data, and the device does not accept writing them back, so they are never
part of a restorable scene. You can still archive them read-only with
`backup --preset` (writing `P001.json` ... `P100.json`), for reference.
====

[[sec-storage]]
=== Where files are stored

[cols="1,3",options="header"]
|===
| What | Location
| Single patches and per-patch backups | `~/.config/rackctl/gx700/patches/<name>.json`
| Scenes (whole-device snapshots)       | `~/.config/rackctl/gx700/scenes/<name>.json`
|===

(The exact base follows the XDG config directory; `~/.config` is the default.)

[[sec-scene-save]]
=== Saving a scene

Capture all 100 user patches into one named file:

[source,console]
----
$ rackctl-gx700 --port hw:1,0 scene save "before-gig"
U001  N RODGERS?
U002  CLEAN CUTTIN
...
U100  EXP DlyTIME
saved scene "before-gig" (100 patches) to /home/you/.config/rackctl/gx700/scenes/before-gig.json
----

This reads every user patch in turn, so it takes a couple of minutes (each patch is
a full read, deliberately paced so it does not overrun the MIDI interface). Leave
the interface to it until it reports the saved file.

[[sec-scene-restore]]
=== Restoring a scene

Write a saved scene back to the device. This *overwrites every user patch*, so it
is destructive and requires `--yes` to proceed -- and, because it stores to the
user bank, the unit must be in BULK LOAD mode (see <<sec-bulk-load>>):

[source,console]
----
# On the GX-700: TUNER/UTILITY -> "MIDI BULK LOAD" ("Waiting...")
$ rackctl-gx700 --port hw:1,0 scene restore "before-gig"
error: restoring scene "before-gig" overwrites 100 user patches on the device; re-run with --yes to confirm

$ rackctl-gx700 --port hw:1,0 scene restore "before-gig" --yes
U001  N RODGERS?
...
restored scene "before-gig" (100 patches) to the device
# then press PLAY on the unit to leave BULK LOAD mode
----

If the unit is not in BULK LOAD mode, the restore stops after the first patch with
a clear error rather than silently writing 100 patches that do not take.

[WARNING]
====
`scene restore --yes` replaces all 100 user patches on the unit with the ones in
the file. Anything currently in those slots that you have not saved is lost. If in
doubt, `scene save` the current state under a new name first.
====

[[sec-scene-list]]
=== Listing scenes

List the scenes you have saved (no device needed):

[source,console]
----
$ rackctl-gx700 scene list
before-gig
studio-2026-06
----

[[sec-single-patch]]
=== Saving and loading a single patch

To keep just one sound, save the current sound or a numbered slot, and load it back
to the current sound or to a user slot:

[source,console]
----
$ rackctl-gx700 --port hw:1,0 save "lead-tone" --patch 42   # save user patch 42
# to write a slot, put the GX-700 in BULK LOAD mode first (see above)
$ rackctl-gx700 --port hw:1,0 load "lead-tone" --to-patch 7 # write it into user slot 7 (destructive)
----

Without `--patch` / `--to-patch`, `save` captures and `load` writes the *current
sound* (the temporary buffer) instead of a stored slot. Writing to the current
sound needs no special mode; writing to a stored slot with `--to-patch` overwrites
that patch and requires BULK LOAD mode (<<sec-bulk-load>>).

[[sec-backup]]
=== Backing up every patch as separate files

`backup` reads the whole user bank and writes each patch as its own file in the
patch library, so you can list, inspect, or restore any single one by name:

[source,console]
----
$ rackctl-gx700 --port hw:1,0 backup            # writes U001.json .. U100.json
$ rackctl-gx700 --port hw:1,0 backup --preset   # writes P001.json .. P100.json (factory, read-only)
----

Because these land in the patch library, the other commands see them immediately:

[source,console]
----
$ rackctl-gx700 patches --disk        # list what you have on disk
$ rackctl-gx700 dump --file U042       # show one patch in readable form (no device needed)
$ rackctl-gx700 --port hw:1,0 load U042 --to-patch 42   # restore one patch (BULK LOAD mode)
----

Use `backup` when you want to cherry-pick individual patches; use `scene save` when
you want one portable file that captures the device as a whole.

[[sec-backup-tips]]
=== Tips and timing

* *One command at a time.* Do not run another GX-700 command while a backup,
  scene save, or restore is in progress (see the note in <<sec-connecting>>).
* *Allow a few minutes.* A full bank is 100 patches, each a complete read or write,
  paced to keep the interface stable. Both directions report progress patch by
  patch and a summary line at the end.
* *Files are plain JSON.* Copy them, keep them in version control, or move them
  between machines. A scene file is self-contained; a patch/backup file is one
  patch.
* *Try it without hardware.* Add `--mock` (and drop `--port`) to exercise any of
  these commands against an in-memory device and see exactly which files are
  written.

[[sec-monitor]]
== Monitoring MIDI

Two commands print what arrives on the MIDI port. They are diagnostic aids -- for
confirming the cabling and for seeing what the unit sends. Both need `--port` and
run until you interrupt them with kbd:[Ctrl+C].

`monitor` decodes every incoming MIDI message to one readable line:

[source,console]
----
$ rackctl-gx700 --port hw:1,0 monitor
ch1  Program     6
ch1  CC          ctrl 7 val 100
SysEx [F0 41 00 79 12 00 00 04 02 64 ...]
----

It decodes note on/off, control change, program change, pitch bend, aftertouch, and
system real-time messages, and shows System Exclusive as hex.

`recv` prints only incoming System Exclusive messages, as raw hex -- a
reverse-engineering aid for reading exactly what the unit emits:

[source,console]
----
$ rackctl-gx700 --port hw:1,0 recv
F0 41 00 79 12 00 00 04 01 03 ...
----

Use these to confirm the *return* path works (GX-700 MIDI OUT -> interface MIDI IN)
and to capture the patch data the unit streams.

[NOTE]
====
The GX-700 does *not* transmit individual parameter edits over MIDI -- it transmits
only when it sends a bulk dump (for example, in reply to a read). So turning a knob
on the front panel shows nothing in `monitor`, while reading a patch shows the whole
dump. See `docs/gx700-sysex-protocol.adoc` for the details.
====

`monitor` and `recv` hold the port like any other command, so the one-program-at-a-
time rule (<<sec-connecting>>) applies: do not run them alongside another GX-700
command.

[[sec-cli-ref-global]]
=== Global options

[cols="1,3",options="header"]
|===
| Option | Meaning
| `--port <hw:CARD,DEV>` | The ALSA rawmidi port of the USB-MIDI interface (see `ports`). Required by every command that talks to the hardware.
| `--mock`               | Use an in-memory mock device instead of hardware; use this *instead of* `--port`.
| `-h`, `--help`         | Print help (top level, or for a `<command>`).
| `-V`, `--version`      | Print the version.
|===

A command that talks to the device needs exactly one of `--port` or `--mock`.

[[sec-cli-ref-commands]]
=== Commands

The *Needs* column shows what a command requires: _device_ (a unit via `--port`, or
`--mock`), _disk_ (only local files), or _none_ (built-in, no device or files).

[cols="2,4,1",options="header"]
|===
| Command | Summary | Needs
| `ports`                          | List the ALSA rawmidi ports on the system. | none
| `list`                           | List every parameter (key, block, kind, address). | none
| `info <param>`                   | Show one parameter's block, kind, range, and enum values. | none
| `get <param>`                    | Read a parameter's current value. | device
| `set <param> <value>`            | Write a value to a parameter (current sound). | device
| `edit <name> <key> <value>`      | Set one parameter by key in a saved patch file (via the typed model). | disk
| `dump [--patch N \| --file NAME] [--json]` | Print a patch: readable decode, or `--json` for the typed block-grouped model. | device / disk
| `save <name> [--patch N]`        | Save a patch (current sound, or slot `N`) to disk. | device
| `backup [--preset]`              | Save a whole bank as per-patch files. | device
| `scene save <name>`              | Save all 100 user patches as one scene file. | device
| `scene restore <name> --yes`     | Restore a scene to the device (overwrites all user patches). | device
| `scene list`                     | List scenes saved on disk. | disk
| `load <name> [--to-patch N] [--json]` | Load a saved patch (or, with `--json`, a typed-JSON file path) to the current sound or slot `N`. | device
| `copy <from> <to>`               | Copy a stored patch to another user slot on the device (overwrites). | device
| `chain <name> [--set …]`         | Show or reorder the signal chain (block order) of a saved patch. | disk
| `select <n>`                     | Recall a patch by Program Change `n` (0--127). | device
| `preview <slot>`                 | Write a stored patch into the active sound to hear it, without storing. | device
| `patches [--preset] [--disk]`    | List patch slots (device user/preset bank, or saved on disk). | device / disk
| `recv`                           | Print incoming SysEx as hex. | device
| `monitor`                        | Print every incoming MIDI message, decoded. | device
| `completions <shell>`            | Print a shell completion script. | none
|===

[[sec-troubleshooting]]
== Troubleshooting

`ports` shows nothing / `no rawmidi ports found`::
No USB-MIDI interface is visible to ALSA. Check it is connected and recognised
(`arecord -l` / `amidi -l` should list it), then re-run `ports`.

`MIDI port hw:1,0 is in use by another rackctl process`::
Another `rackctl-gx700` already holds the port. Only one program may use the
interface at a time (see <<sec-connecting>>). Wait for the other command -- a
backup, scene save, or restore can take a few minutes -- to finish.

`timed out waiting for a device reply`::
The unit did not answer. Check, in order: both MIDI cables (interface OUT ->
GX-700 IN *and* GX-700 OUT -> interface IN); that `--port` is the right interface;
that no other program is reading the port; that the GX-700 is powered on. If the
interface itself has stalled after heavy use, power-cycle it.

`set` changes nothing audible::
The GX-700 receives on its MIDI channel (omni by default); confirm the channel /
device id if you have changed it. Remember that `set` edits the *current sound*, so
make sure that is what you are listening to (recall a patch with `select` first).

`copy` / `load --to-patch` / `scene restore` says the patch was not stored::
The GX-700 only accepts patch-memory writes in BULK LOAD mode. On the unit, press
TUNER/UTILITY, select "MIDI BULK LOAD" ("Waiting..."), then re-run the command, and
press PLAY when done. See <<sec-bulk-load>>.

Listing or backing up a bank is slow::
Expected. Each patch is a complete read, and reads are paced to keep the interface
stable; a full bank is therefore a couple of minutes.

Intermittent timeouts or garbled reads::
Almost always two programs using the port at once -- the device's reply gets split
between them. Make sure only one `rackctl-gx700` (or other MIDI tool) is using the
interface.

[[sec-glossary]]
== Glossary

Patch:: One complete GX-700 sound -- every effect block and its settings, plus a
name and the effect-chain order. The unit holds 200.

Current sound:: The temporary buffer -- what the unit is playing right now. `get`,
`set`, and `dump` (with no slot) act on it.

User / preset bank:: The 100 user patches (`U001`..`U100`), which you can overwrite,
and the 100 factory preset patches (`P001`..`P100`), which are read-only.

Scene:: A named snapshot of all 100 user patches, saved as a single file.

System Exclusive (SysEx):: The category of MIDI message the GX-700 is controlled
with. DT1 is the Roland "write" command and RQ1 the "read" request.

Program Change:: The MIDI message that recalls a patch memory.

USB-MIDI interface:: The hardware that bridges the computer's USB to the GX-700's
5-pin DIN MIDI.
