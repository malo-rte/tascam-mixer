= Avid/Digidesign Eleven Rack — MIDI System Exclusive protocol reference
:toc: left
:icons: font
:source-highlighter: rouge
:experimental:

This document describes how the Avid (Digidesign) Eleven Rack guitar amp/effects
processor is controlled over MIDI, as it will be implemented by the planned
`rackctl-eleven` crate. It records what the device actually does -- confirmed
against a real unit -- and is the reference for the parameter catalog.

Unlike the GX-700, there is *no* published MIDI implementation. Avid never
documented the protocol the standalone "Eleven Rack Editor" used, and the editor
was discontinued (it required Pro Tools and broke on modern 64-bit hosts). Every
fact below was *reverse-engineered from hardware* (a unit running firmware
`0157`, reached over ALSA raw MIDI) and, where useful, cross-checked against the
community's prior art (USB sniffing with Wireshark / MIDI Monitor, and Max/MSP and
Python scripts that map parameters to CC). The device is the final word.

NOTE: Eleven Rack, Digidesign and Avid are trademarks of Avid Technology, Inc.
This is an independent, unofficial project, not affiliated with Avid; the names
identify the hardware these tools control.

== At a glance

* The Eleven Rack is a *rig-based* guitar amp/effects modeller and an 8x8 USB 2.0
  audio interface. Editing is done entirely over MIDI System Exclusive; there is no
  ALSA mixer surface for the DSP parameters.
* Control is *address-mapped System Exclusive* -- the same _shape_ as the GX-700's
  Roland scheme, but with Digidesign framing. Every parameter lives at a multi-byte
  address; you read it with a request and the unit answers with a report.
* The unit does *not* volunteer its state. A rig change does not stream the new
  rig -- reads must be *polled*, one address (or a batch of requests) at a time.
* Turning a front-panel knob emits an unsolicited change report for that one
  parameter, which is how addresses are discovered and labelled.

== Transport (ALSA)

The unit enumerates as a USB-MIDI device (ALSA card, two raw-MIDI ports):

[cols="2,3",options="header"]
|===
| Port | Use
| `hw:<card>,0,0` "Eleven Rack Rig" | Editor / parameter SysEx. *This is the port.*
| `hw:<card>,0,1` "Eleven Rack External" | DIN MIDI thru / external pedal. Silent in testing.
|===

The Rig port is silent when idle; all traffic is request/response or change-driven.
This reuses the byte-level `rackctl-midi` link unchanged.

== USB topology

For reference, the device's full USB descriptor (`0dba:b011`, USB 2.0 high-speed,
`bDeviceClass 0xEF` multi-interface, one configuration, five interfaces). Parsed from
the kernel's config descriptor, including alternate settings:

[cols="1,2,1,1,1,1,3",options="header"]
|===
| Interface | Class | Alt | Endpoint | Type | maxpkt | Purpose
| IF0 | DFU `fe.01.00` | 0 | -- (EP0) | -- | -- | Firmware upgrade
| IF1 | vendor `ff.01.20` | 0 | -- (EP0) | -- | -- | Vendor control / management
| IF2 | *MIDIStreaming* `01.03` | 0 | `0x02` | Bulk OUT | 512 | *MIDI / SysEx out*
| | | | `0x82` | Bulk IN | 512 | *MIDI / SysEx in*
| IF3 | vendor `ff.02.20` | 0 / *1* | `0x03` | Isoc async OUT | 416 | Audio playback (host -> unit)
| IF4 | vendor `ff.02.20` | 0 / *1* | `0x83` | Isoc async IN | 416 | Audio capture (unit -> host)
|===

What this means for the project:

* *Control is IF2* -- class-compliant MIDIStreaming over bulk endpoints
  `0x02`/`0x82`. That is why `snd-usb-audio` binds it and exposes the ALSA raw-MIDI
  ports; all the SysEx in this document flows through here. Nothing proprietary is
  needed for control.
* *The 8x8 audio is not class-compliant.* The streams are plain *vendor-specific
  isochronous* endpoints (IF3/IF4 alt 1), with no USB-Audio `AudioStreaming`
  descriptors, so the kernel will not drive them as a sound card. This is why the
  Eleven Rack needs Avid's proprietary driver for audio on a host, and it is why
  the audio side is *out of scope* (see `eleven-rack-roadmap.adoc`).
* *IF0 (DFU)* is firmware flashing and *IF1 (vendor control)* is what the legacy
  editor used for some control-transfer traffic; neither is needed, since the editor
  protocol we target runs over MIDI/SysEx on IF2.

== Message framing

All device messages share one envelope:

----
F0  13  0B  0F  <opcode>  <address...>  <value...>  F7
│   │   │   │
│   │   │   └ device id   0x0F
│   │   └ model           0x0B  (Eleven Rack)
│   └ manufacturer ID     0x13  (Digidesign)
└ SysEx start                              ... F7 = SysEx end
----

CAUTION: `0x13` as a one-byte MIDI manufacturer ID is what this unit emits; it is
recorded here as an observed fact, not from a registry.

== Device identity

The unit answers the *Universal* Identity Request, which is the safe first probe:

[cols="1,3"]
|===
| Request (universal) | `F0 7E 7F 06 01 F7`
| Reply               | `F0 7E 0F 06 02 13 0B 00 01 00 30 31 35 37 F7`
|===

Decoded reply: device id `0x0F`; manufacturer `0x13`; family `0x000B` (LSB-first
`0B 00`); model `0x0001` (`01 00`); version = ASCII `"0157"` (`30 31 35 37`).

== Opcodes

[cols="1,2,4",options="header"]
|===
| Opcode | Name | Direction / meaning
| `0x01` | Read request | host -> unit: `F0 13 0B 0F 01 <addr> F7`. Ask for one address.
| `0x12` | Read reply | unit -> host: `F0 13 0B 0F 12 <addr> <value> F7`. Solicited answer to `0x01`.
| `0x02` | Change report | unit -> host: `F0 13 0B 0F 02 <addr> <value> F7`. Unsolicited, when a knob moves.
| `0x00` | Write (set) | host -> unit: `F0 13 0B 0F 00 <addr> <value> F7`. Writes a parameter (3-byte addr) or a whole block (1-byte addr). Confirmed by set + read-back. Affects the *edit buffer* only (see note).
|===

.Confirmed read round-trip (amp Gain)
----
->  F0 13 0B 0F 01 11 21 0D F7
<-  F0 13 0B 0F 12 11 21 0D 6D 00 00 00 10 F7
----

The read request's *address width* selects the granularity:

* *3-byte address* (`11 <model> <index>`) -> a single parameter (5-byte value).
* *1-byte address* (`<block_id>`) -> the *whole block*, returned as one
  `12 <block_id> <payload> F7` — this is the *on-demand bulk dump*. No boot or
  handshake is needed; it answers whenever the unit is awake.

.Confirmed block dumps (on-demand, `01 <block_id>`)
[cols="1,3",options="header"]
|===
| Request | Reply payload
| `01 05` | the rig *name* as ASCII (e.g. `"A is A 800"`)
| `01 08` | the FX-chain config (`… 7F 00 00 64 …`, the `.tfx` `FXc` slots)
| `01 21` | the amp model's parameter table (see below)
| `01 01` | a large (~1.3 KB) packed full-patch blob (unit-internal format)
|===

A parameter-table block (e.g. `0x21`) is `<count>` then `count` 3-byte records
`[value][index][target]`: `value` is the current 0..127 setting, `index` is the knob
id (the byte2 of a live `11 <model> <index>` read), and *`target` is the indirection
pointer* — it identifies the actual parameter slot in the patch the index resolves to.
This is the "index into a table that points at the real parameter" the device uses.

*Addressing amp parameters (hardware-verified, fw `0157`).* The `index` is *unstable*
— it is reassigned on every reload, so it must be read live. Address a parameter by its
stable `target`: read the table block (`01 21`), find the record with that target, and
use its current `index`. The CLI does exactly this — `scan amp` dumps the table
(target / value / live index) and `get <target>` reads a value.

=== Live parameter access (decoded from the editor USB capture)

Decoding the Windows editor's USB-MIDI traffic (`11/*.pcapng`, usbmon; split by
direction and de-framed from the 4-byte USB-MIDI packets) settles how parameters are
read, set and reported:

* *Set = opcode `0x00`* (host→unit). The editor sets a parameter with the *block-write*
  opcode, e.g. `00 11 44 02 <index> <value-word> F7`. (Our earlier per-knob `aseqdump`
  logs showed `0x02` — but those were the unit's *reports*, not the editor's sets.)
* *Report = opcode `0x02`* (unit→host). The unit echoes each change with `0x02` and the
  same address, e.g. `02 11 44 02 <index> <value-word> F7`. Value words differ only in
  the trailing tag byte (`…00` written vs `…10` reported).
* *Read = opcode `0x01`* → reply `0x12`, as before.

*Parameter address: `11 <block> <sub> <index>`.* The `<block>` is the *`.tfx` block id*
— the editor reads `11 42 01`, `11 43 01`, `11 44 0n`, `11 45 0n`, … `11 49 01` (the
*amp* — `0x49` is the `.tfx` amp block), `11 4A/4B/4C` (fx-loop / delay / reverb). So the
live SysEx parameter space is addressed by the *same block ids as the `.tfx` file* — the
missing link between the two. `01 11 <block> <sub>` reads a sub-block's parameter table;
`00 11 <block> <sub> <index> <value>` writes one.

*Store* (unchanged) follows the same session: `00 05 <name>` then `00 03/04/02 <slot>`.

*Amp knob address (decoded from a rich capture — physical + software knob turns).* A
larger capture (turning knobs on the rack, then in the editor, then saving) shows the
amp knobs are set/reported at `11 <model-block> <sub>` — *three* address bytes, no
trailing index. `<model-block>` is the loaded amp model's block (`0x3D` for the captured
model); `<sub>` is the knob:

[cols="1,1",options="header"]
|===
| `<sub>` | Amp knob
| `07` | Gain
| `09` | Master
| `0A` | Treble
| `0B` | Mid
| `0C` | Bass
| `0D` | Presence
| `0E` | Bright
|===

So amp Gain is `00 11 3D 07 <value-word>` (set) / `02 11 3D 07 <value-word>` (report).

*Value word — two forms.* The 5-byte value word differs by *source*:

* A *physical* knob turn reports the **integer** form `<v> 00 00 00 10` (`v` = `0..=127`,
  trailing tag `0x10`). This is exactly the form our `write_param` builds.
* The *editor* sets a knob with a **packed float** (full precision), e.g.
  `40 28 7A 70 0A` — a 32-bit float across the 5 seven-bit bytes, with a *variable*
  trailing tag (`0A`/`04`/`06`/`0F`, not `0x10`).

*Hardware findings (why host-set still does not take).* Tested against the unit with
`amidi` (which sends arbitrary SysEx) plus capture/diff:

* A `0x00` *block* write reaches and changes the unit — e.g. `00 05 <name>` renames the
  edit buffer, and the aggregate `0x01` write loads a full patch. So transport and the
  write opcode are correct.
* A `0x00` *parameter* set does **not** take on our patches — not with the integer form,
  not replaying the editor's *exact* float bytes, at block `0x3D` or swept across
  `0x30..0x3F`. Nothing changed.
* Reading the parameter namespace the editor uses returns **nothing** on our patches:
  `01 11 49 01` (tfx amp), `01 11 3D <sub>`, and a sweep of `01 11 <block> 01` across
  `0x38..0x60` all got no reply — while `01 05`, `01 04 <slot>`, and the `01 21` amp
  table *do* reply.
* There is *no* special handshake opcode: before its first parameter set the editor only
  reads the patch (hundreds of `0x01` reads) and issues `00 02 <slot>` recalls; the store
  is the usual `00 05` / `00 03/04/02`.

So the `11 <block> <sub>` live-parameter namespace is *activated by the editor's full
session/model context*, not by a bare Program Change. This was confirmed against the
capture's *exact* patch: it was made on User A1 (`Big Blue`, slot 0), and even with
`Big Blue` loaded — and after an editor-style recall `00 02 00 00` — the unit still
returns nothing for `01 11 3D 07` and ignores a replayed set. So it is *not* a
patch/model mismatch; it is session state.

The likely key is the model-definition area the editor reads at startup: `01 20 <n>`
returns model/def records (`01 20 00` → `12 20 00 "DigiElvn…Eleven"`), and `01 22` is the
device-info register (`0x41`).

*Init-replay attempts (both negative).* Extracted the editor's exact startup stream (the
535 messages before its first parameter set — 531 reads + four `00 02 <slot>` recalls) and
replayed it two ways:

. Over `amidi` (each call a fresh port open/close). `01 11 3D 07` still returned nothing
  and the set was ignored.
. Over a *single connection* — a purpose-built replayer that opens the rawmidi port once,
  sends the whole init (paced, draining the unit's replies), then probes and sets without
  closing. Same result: `01 11 3D 07` / `01 11 49 01` still got *no reply*, and the set did
  not take. (The amp table `0x21` and aggregate `0x01` do change after the replay, but a
  *no-set control* changes them identically — it is just `0x21`'s volatile index
  reshuffling from the init reads/recalls, not the set.)

So per-connection state is *not* the gate either. USB control transfers are ruled out (the
capture has none). A second, shorter capture ruled out two more theories: the only
non-Digidesign SysEx the editor sends is the *standard Identity Request* (`F0 7E 7F 06 01`
→ reply `F0 7E 0F 06 02 13 0B 00 01 00 30 31 35 37`), i.e. nothing proprietary that could
"unlock" editing; and edit mode is *not* a per-set keep-alive — that session has a run of
157 consecutive parameter sets with no interleaved reads, so once editing works you can
blast sets.

The editor clearly does something more — precise request/response timing/order, or state
we have not isolated — that a bare Program Change plus a replay of the captured MIDI does
not reproduce (notably: on our sessions the unit never *replies* to `01 11 <block> <sub>`
reads, which the editor's session always gets answered). Host-initiated parameter set
remains *unresolved*; cracking it likely needs diffing the editor's *live* traffic against
a working set in real time, or the editor DLL's logic. Live parameter changes go through
MIDI CC (`Bn <cc> <v>`) and a full patch change through the aggregate `0x01` write — both
hardware-verified.

Requesting every populated block id (a *small paced batch* — not a flood) reads the
whole patch without per-parameter polling. Other blocks seen on one patch: `0x05` name,
`0x08` FX-chain routing, `0x07` a small config word, `0x01` the packed blob, and several
single-value blocks (per-slot model ids / settings).

IMPORTANT: these blocks reflect the *currently selected* patch — they are not addressed
by patch number. The name at `01 05` follows a Program Change (`"A is A 800"` ->
`"Diezel  Einstein"` after `C0 0A`), and the `01 01` blob changes size per patch. So the
*whole patch library* is backed up by *iterating Program Change over every patch and
dumping each* (read `01 01` + `01 05` per patch) — the "dump them one at a time" method.
Rig selection is `CC 32` bank (`1`=Factory, `0`=User) then Program Change.

Batching many `0x01` requests then collecting the `0x12` replies enumerates an
address range in one pass -- the basis for a `scan`/`dump` command.

== Addressing

The address is *multi-byte* (three bytes seen). It behaves like the GX-700's
section/block/parameter scheme: a leading block selector, then a contiguous array
of parameter offsets.

The three bytes act as *section / model-table / index*. byte1 selects the *active
model's parameter table* (hence it is model-dependent — the DC Modern amp answers at
`0x3D`, an earlier rig's amp at `0x21`), and byte2 is an *index into that table* that
points at the actual parameter in the patch. The catalog is therefore keyed by
`(model-table id, index)`; if the table itself proves readable, a whole model's
parameters could be enumerated at once rather than swept one knob at a time.

Mapped by scanning (firmware `0157`):

* *Section `0x11` is the current rig's parameters.* Under it, byte1 enumerates
  *~13 blocks* at `0x1E`..`0x2A` (the rig's effect blocks), each a 128-entry
  parameter array (byte2 `0x00`..`0x7F`). Most entries sit at a centre default
  (`40 00 00 00 00` = 64); the populated ones are the live parameters. Block
  `11 21` is the densest (~20 active) and holds the amp — amp *Gain* = `11 21 0D`.
* *Other sections carry strings/metadata* — e.g. amp/cab *model names* read as
  ASCII in sections around `0x05` / `0x20` (`"Marshal…"`, `"DigiDy…"`), and the rig
  name elsewhere.

[cols="2,3",options="header"]
|===
| Address | Meaning
| `11 21 0D` | Amp Gain (front-panel knob) — anchor, confirmed by sweep
| `11 1E..2A xx` | Section `0x11` rig blocks (byte1 = block, byte2 = parameter)
|===

WARNING: The block->effect assignment (which of `0x1E`..`0x2A` is amp/cab/stomp/mod/
delay/reverb/FX-loop/master) and the per-parameter value scaling are still being
decoded — by correlating live reads with the `.tfx` rig schema
(`eleven-rack-rig-format.adoc`) and the User Guide's MIDI CC reference (the parameter
names and value semantics).

== Value encoding

Each value is *five MIDI bytes*: a 32-bit word packed as five little-endian 7-bit
groups (`b0` = least-significant 7 bits). Reconstruction:

[source]
----
value = b0 | (b1 << 7) | (b2 << 14) | (b3 << 21) | (b4 << 28)
----

Sanity checks from a live amp-block scan:

[cols="2,2,2",options="header"]
|===
| Address | Bytes | Decoded
| `11 21 00` | `40 00 00 00 00` | `0x40` = 64
| `11 21 02` | `7F 7F 7F 7F 0F` | `0xFFFFFFFF`
| `11 21 0D` | `6D 00 00 00 10` | low 7 bits = `0x6D` (Gain), plus a high tag bit
|===

For the front-panel controls mapped so far the value lives in *b0* alone, with the
upper bytes a constant type tag (`00 00 00 10`):

* *Knob* parameters: `b0` = `0..127` (e.g. amp Gain sweeps `0x00..0x7F`).
* *Switch / bypass* parameters: `b0` = `0x3F` (off) / `0x40` (on) — the same
  `0-63 = off, 64-127 = on` split the User Guide's CC reference uses.

Some other parameters use the full 32-bit word (multi-field or float-like); those
are not a plain IEEE-754 float in this bit order (a naive read yields NaN) and their
scaling is still being decoded. The on-disk `.tfx` stores the *effect* parameters as
32-bit floats (see `eleven-rack-rig-format.adoc`).

[TIP]
====
The reliable, safe way to identify a parameter's address is *passive listening* —
it sends no read requests, so it can never overrun or wedge the unit:

----
aseqdump -p <client>:0        # the "Eleven Rack Rig" sequencer port
----

Turn one control; the unit emits `F0 13 0B 0F 02 <addr> <value> F7` and the address
is that control's. Sweeping the knob min->max also gives its range. (Polling the unit
with read requests is fine in small bursts but a sustained flood makes its editor
parser go dormant — see the caution below.)
====

CAUTION: Do *not* flood the unit with read requests. A large sustained scan (thousands
of `0x01` requests) overruns its MIDI parser and the editor interface goes *dormant* —
it keeps answering the Universal Identity request but stops answering parameter reads,
and a power-cycle may not revive it (re-enumerating the USB cable does). Keep reads
paced and in small bursts, and prefer passive listening for discovery.

== Parameter catalog

The Eleven Rack has *two* parameter namespaces, and the catalog spans both:

. *MIDI CC numbers* (remote control) — the authoritative *names, per-model
  structure and value semantics* of every control, transcribed from the User
  Guide's Chapter 11 into `crates/eleven-model/src/param.rs` (the `param` module:
  26 amp models, ~22 effects, the shared stepped-value tables, the `General`/`Misc`
  controls, plus the amp section's 7 cabinet and 7 microphone models and the mic
  on/off-axis position from chapter 3). This is the canonical parameter list and
  drives the CLI `list` / `info` commands. See *MIDI CC reference* below.
+
The base User Guide (v8.0.4) *predates the Expansion Pack (firmware 2.x)*, so its
printed CC chart lacks the added amps and effects — but the unit's *on-device* MIDI
CC Reference (User Options) is firmware-generated and includes them. It was
transcribed from photographs of the screen: all *effects* (base + the Expansion
Multi Chorus, Parametric EQ, Dyn3 Compressor, White Boost, DC Distortion, Dyn Delay)
are fully catalogued with a **CC per slot** (the mapping is not a positional formula
— Parametric EQ skips a slot and Multi Chorus's `Lo Cut`/`Width` share one CC across
slots — so [`FxParam`] stores the CC for each of an effect's [`Effect::slots`]). The
Expansion *amp* models are added too (J45, Black SR, Black Vib, Blue Line Bass, MS-30,
RB-01b, plus the family members Plexiglas Vari, Plexiglas 50W, DC Bass, Lux Mini). The
on-device CC Reference groups amps by shared-CC *family* (e.g. `PLEXIGLAS/J45`, `DC
MOD/VIN/BASS`) rather than listing every individual model, so those last four names are
*inferred from the family groupings* and each reuses its family's CC layout — worth a
final check against the device's amp-select list.
. *SysEx addresses* (`11 <block> <index>`) — what the editor actually reads/writes.
  The block byte is *model/slot-specific* and the index is a small *sequential*
  offset; these are mapped by live capture (below), **not** derivable from the CC
  numbers. The two namespaces are unrelated: for amp *DC Modern* the captured
  indices are Gain `07`, Master `09`, … Presence `0D`, whereas its CCs are Gain 13,
  Master 10, Presence 21. (The lone `crate::AMP_GAIN` = `11 21 0D` matches CC 13
  only by coincidence on that one model.)

So `param.rs` answers "*what* are the parameters, their ranges and meanings"; the
capture tables below answer "*where* a given model writes them". Binding the two
per model (CC/name ↔ internal index, and the effect-slot block ids) is the
remaining hardware-mapping work.

[NOTE]
====
The installed Windows *Eleven Rack Editor* was mined as a possible source for the
parameter metadata and is a *dead end*: it is a Pro Tools-derived application whose
extractable strings are all Pro Tools UI, debug symbols (`DigiTrace…`), or skinned
UI image-asset names (`.idb`/`.vdb`). It reads the *model names* from the device
(block `0x20`) at startup and renders parameter UIs from binary resources — there is
no extractable per-model parameter table in it. The User Guide's Chapter 11 chart is
the real source.
====

== Rig selection

Front-panel rig changes do *not* emit a full parameter dump (reads must be polled).

Rig *selection* is standard MIDI, per the User Guide (Chapter 11, "Controlling
Eleven Rack with MIDI"): a *Bank Select* on *CC 32* (value `1` = Factory rigs,
`0` = User rigs) *precedes* a *Program Change* that selects the rig within the bank.
This is the foot-controller scheme (FCB1010 / Ground Control Pro). Whether a
whole-rig *bulk* read/write exists over the editor SysEx is still open.

== Storing a patch (save the edit buffer to a slot)

A write (`0x00`) changes the *edit buffer* — the current sound — but does not persist;
reselecting the patch reloads the slot. To *save* to a slot, issue three further `0x00`
writes to the *directory/commit* blocks `0x03`, `0x04`, `0x02` (these never appear in a
normal read dump). Sequence saving the current rig as `"ABCDE"` to slot 6:

----
00 05 41 42 43 44 45 00            ; set edit-buffer name (block 0x05), NUL-terminated
00 03 <lo> <hi>                    ; save the edit-buffer CONTENT to the slot (LITTLE-endian!)
00 04 <hi> <lo> 41 42 43 44 45 00  ; directory entry (name) for the slot (big-endian)
00 02 <hi> <lo>                    ; commit
----

CAUTION: the two slot-address bytes are ordered *differently per message*. The content
save `00 03` takes the slot *low byte first* (`<lo> <hi>`); the directory `00 04` and
commit `00 02` take it *high byte first* (`<hi> <lo>`). This was found the hard way:
sending `00 03` big-endian (`00 03 00 06` for slot 6) made the unit read the slot as
*0* and write *truncated* content there — so **every store silently overwrote User
slot 0**. Hardware-verified across slots 0/4/6: with `00 03 <lo> <hi>` the store writes
exactly the target slot (slot 0 untouched) and persists the *full* 1235-byte content
across a reselect.

The `0x00` writes affect the edit buffer; the store commits it. So a *full* patch copy is
simply: get the source into the edit buffer (select a slot, or write its blocks), then run
this sequence for the target slot — no per-block reconstruction needed. (The unit echoes
each write as a `0x02` report; read back to verify.)

== Restoring a patch (write the edit buffer, then store)

Implemented as a *device-faithful* round-trip that avoids decoding per-parameter value
scaling. A `PatchBackup` (the `rackctl-eleven save` command) snapshots a patch as the raw
payloads the unit returned for each content block, including the *aggregate* block `0x01`
— the whole packed patch in one blob. This management logic lives in `rackctl-eleven-lib`'s
`manage` module, shared by the CLI and the GUI.

*Primary path — write the aggregate.* The aggregate block `0x01` is *writable*: writing it
whole into the edit buffer (`00 01 <1235-byte image>`) and then storing loads the patch *in
full, byte-for-byte* — no per-block reconstruction, no value scaling. *Hardware-confirmed*
(firmware `0157`): after `load`, the target slot's re-read `0x01` is byte-identical to the
source, and slot 0 is untouched. So `load`/`copy` just write `0x01` + store, and verify by
comparing the re-read `0x01`. (This is also the delivery mechanism for writing an imported
`.tfx` once a valid `0x01` image can be built from it.)

*Fallback — per-block replay.* Only a backup that carries no `0x01` (an old/partial capture)
replays the individual restorable blocks. The action depends on the block kind:

* *Parameter-table blocks* (the amp `0x21`): a whole-block write does *not* take — the unit
  rebuilds the table itself. They are replayed *per parameter*. Crucially, the record
  *index* is **not stable**: the same parameter appears at a different `index` after a reload
  (a captured amp had its knobs at indices `0x4c..0x56`; after restore they were `0x1e..0x28`).
  The *`target`* byte is the **stable logical id**. So restore re-reads the *live* table,
  maps each captured value's `target` to its current `index`, and writes `00 11 <block>
  <live-index> <value>`. Verify likewise compares values keyed by `target`, not by raw bytes.
* *Safe flat blocks* — *only* the name `0x05` and FX-chain `0x08` — are written whole
  (`00 <block> <payload>`).
* *Everything else is skipped.*

[CAUTION]
====
*Do not blindly whole-block-write the captured range.* Many blocks a capture turns up are
*device/system* state, **not** patch content: `0x22` is the unit's *device-info* register
(read at startup as `01 22`), the single-byte `0x30..0x3f` blocks are model selectors / system
flags, and `0x34` is the store/revision counter (it ticks `127 → 126 …` per store). An early
restore wrote *all* of them; writing the `0x22` device-info register put the unit into *Retail
Demo Mode* — a dimmed display (bright at boot, fades in 2-3 s; **not** a backlight fault) and a
*blocked* "Reset Memory". Recovery is a *hardware total reset*: hold *REV + MOD + FX LOOP* while
powering on (this bypasses the software menu; `REV + FX LOOP` is the non-destructive flush). So
restore now writes a flat block only if it is on the `SAFE_FLAT_BLOCKS` allowlist (`0x05`,
`0x08`); parameter values still restore fully via the per-parameter path. A *cross-model*
restore (which needs the model-selector blocks set first) is deferred until those system blocks
are individually characterised.
====

[NOTE]
====
*Open:* on the amp table, 8 of 11 parameters restore exactly; three (`target` `0x06`/`0x07`/
`0x08` on the test amp) keep their previous value — a write to their index does not stick,
so they are a distinct *derived/computed* parameter class needing a different write (still to
be characterised; the sustained read/write also eventually wedges the editor parser, so this
needs paced probing on a freshly re-enumerated unit).
====

== Startup sequence and directories

A USB capture of the Windows editor connecting shows it has *no special wake/init* —
it just reads, in order:

. `01 22` -> device info (one byte; `0x41` on the captured unit).
. `01 20 <0x00..0x40>` -> the *model catalog* (65 entries; read-only).
. `01 04 <slot>` -> the *patch directory* for every slot (~208), each entry a flag
  byte + the slot's name.
. then the current patch's blocks (`01 21`, `01 05`, …) and its parameters
  (`01 11 <model> <index>`).

So two read-only directory blocks anchor everything: *`0x20` = the model catalog*
(every amp/effect/input the unit knows, indexed) and *`0x04` = the patch directory*
(every slot's name). Reading `0x04` is a faster way to list the whole library than
iterating Program Change. Block `0x04` is *also* the slot a save targets (see *Storing
a patch*).

== MIDI CC reference (foot-controller control)

Separately from the editor's address-mapped SysEx, the unit exposes many parameters
as MIDI Continuous Controllers for real-time/foot-controller use. *Sending* a channel
message `Bn <cc> <value>` (channel `n`) moves the mapped parameter in the edit buffer —
this is the safe, native remote-control path, and what the CLI `cc` command / the
`rackctl-eleven` `send_cc` use. Block *bypass* CCs: DIST 25, MOD 50, DELAY 28, REVERB
36, FX LOOP 107, WAH 43, AMP 111, FX1 63, FX2 86; globals: Volume Pedal 7, Wah Position
4, Multi-FX 11, Tap Tempo 64.

*Source.* The base User Guide's Chapter 11 chart is *incomplete* — it predates the
Expansion Pack. The authoritative full chart is the unit's own *on-device "MIDI CC
Reference"* screen (User Options), which is firmware-generated and includes the
Expansion amps/effects; it was photographed and transcribed into
`crates/eleven-model/src/param.rs` (the `param` module), surfaced by the CLI `list` /
`info`.

*Per-slot, not positional.* An effect's parameter CCs *differ by slot* (its "as MOD /
as FX1 / as FX2" columns), and the mapping is **not** a positional formula — Parametric
EQ skips a slot position, and Multi Chorus's `Lo Cut`/`Width` share one CC across all
slots. So each `param::FxParam` stores an explicit CC per slot (aligned to
`Effect::slots`, resolved by `Effect::cc_in`), taken verbatim from the on-device chart.

These CC numbers are a *different* namespace from the internal SysEx addresses (see
*Parameter catalog* above); the chart's names, per-model structure and value semantics
(e.g. bypass `0-63 = Off, 64-127 = On`) are authoritative. (See
`eleven-rack-rig-format.adoc` for the rig-file keys, and `docs/external-specs/README.adoc`
for the manual.)
